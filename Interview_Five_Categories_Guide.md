## 第一类：底层原理理解

> **考察重点**：不是回答清楚概念，而是讲清楚方法解决什么问题、存在哪些局限性、有哪些改进方法

### 1.1 RL 后训练的核心设计思想

**面试官可能问**：为什么 LLM 需要 RL 后训练？直接用 SFT 不行吗？

**回答框架**：

**解决的问题**：
- SFT 只能学习"模仿"，无法学习"探索"
- RL 可以让模型发现新的解题路径（如 DeepSeek-R1 的推理能力）
- RL 可以优化人类难以标注的目标（如代码正确性、数学证明）

**局限性**：
- 训练不稳定：RL 的 reward hacking 问题
- 样本效率低：需要大量 rollout 数据
- 超参数敏感：学习率、clip ratio 等需要精细调优

**改进方法**：
- GRPO：去掉 Critic 模型，降低计算成本
- DAPO：动态采样避免同质化数据
- TIS：处理 off-policy 数据

**结合 slime 代码**：
```python
# slime 的设计：统一训练/rollout/数据缓冲路径
# 解决的问题：避免系统变成断开连接的训练器、rollout 服务、Agent 框架
# 设计思想：显式数据流，便于调试和扩展
```

### 1.2 GRPO vs PPO 的设计权衡

**面试官可能问**：GRPO 为什么不需要 Critic？这样设计的优劣是什么？

**回答框架**：

**GRPO 的设计动机**：
- PPO 需要 Critic 模型估计价值函数，但 LLM 的 Critic 模型很难训练
- GRPO 使用组内相对优势，避免了 Critic 的需求
- 计算成本降低约 50%（不需要 Critic 前向传播）

**局限性**：
- 组内相对优势可能不如 Critic 估计准确
- 对 reward 的质量更敏感
- 可能存在 "group collapse"（所有响应都相同）

**改进方向**：
- Reinforce++：加入 baseline 提高稳定性
- GSPO：改进优势估计方法
- 混合方法：对部分样本使用 Critic

**结合 slime 代码**：
```python
# slime 支持多种算法
--advantage-estimator grpo  # 或 ppo, reinforce_plus_plus, gspo
--use-kl-loss  # KL 散度控制，防止策略偏离太远
--kl-loss-coef 0.001  # 为 0 时只作为监控指标
```

### 1.3 Loss Masking 在 Agent 训练中的作用

**面试官可能问**：为什么 Agent 训练需要 Loss Masking？不 mask 会怎样？

**回答框架**：

**解决的问题**：
- Agent 轨迹包含模型生成和工具返回两部分
- 工具返回的内容不是模型生成的，如果参与 loss，模型会学习"复制"工具输出
- 这会导致模型不学习如何正确使用工具，而是学习如何复制结果

**局限性**：
- 需要准确识别哪些 token 是工具返回的
- 多轮交互中，token 边界可能不清晰
- 某些场景下，工具返回的内容也可能有价值（如搜索结果）

**改进方法**：
- 分段训练：将轨迹分成多个段，分别训练
- 优势分配：对不同位置的 token 赋予不同的权重
- TIS：修正 off-policy 数据的分布偏差

**结合 slime 代码**：
```python
# slime 的 loss_mask 机制
sample.loss_mask = [1] * len(model_tokens) + [0] * len(tool_tokens)

# 在 Sample 数据类中
@dataclass
class Sample:
    loss_mask: list[int] | None = None  # 1 = 参与 loss，0 = 不参与
```

### 1.4 权重同步机制的设计权衡

**面试官可能问**：为什么需要 Delta Weight Sync？全量同步有什么问题？

**回答框架**：

**全量同步的问题**：
- 成本与模型大小线性相关
- 对于 355B 模型，每次同步需要传输数百 GB 数据
- 在跨数据中心场景下，网络带宽成为瓶颈

**Delta Sync 的设计**：
- 只传输变化的权重（~3% 密度）
- 使用逐字节比较，无损、dtype 无关
- 支持多种编码方式（indices, deltas, deltas_zstd）

**局限性**：
- 需要额外的 CPU 内存存储快照
- 编码/解码有计算开销
- 在某些场景下，变化的权重可能超过 3%

**改进方向**：
- 自适应编码：根据变化密度选择最优编码
- 压缩算法：进一步减少传输量
- 异步传输：与训练重叠

**结合 slime 代码**：
```python
# slime 的 Delta Sync 实现
# 1. Diff：逐字节比较
current.view(int_dtype) != snapshot.view(int_dtype)
# 2. Encode：编码变化的 (position, value) 对
# 3. 传输：通过 NCCL 或磁盘
# 4. 应用：NaN-masked overwrite
```

### 1.5 动态采样的设计动机

**面试官可能问**：为什么要动态采样？直接采样不行吗？

**回答框架**：

**直接采样的问题**：
- 某些 prompt 可能所有响应的奖励都相同
- 这些"同质化"数据对训练没有帮助
- 浪费计算资源

**动态采样的设计**：
- 采样比实际需要更多的 prompts
- 过滤掉奖励标准差为 0 的样本
- 自动触发新一轮采样补充不足

**局限性**：
- 过滤可能丢弃有价值的样本
- 采样效率可能降低
- 需要精心设计过滤函数

**改进方向**：
- 自适应过滤：根据训练进度调整过滤阈值
- 优先级采样：优先采样有价值的 prompt
- 缓存机制：重用已生成的样本

**结合 slime 代码**：
```python
# slime 的动态采样实现
--over-sampling-batch-size 64
--dynamic-sampling-filter-path slime.rollout.filter_hub.dynamic_sampling_filters.check_reward_nonzero_std

# 过滤函数示例
def check_reward_nonzero_std(args, samples: list[Sample], **kwargs):
    rewards = [sample.get_reward_value(args) for sample in samples]
    keep = torch.tensor(rewards, dtype=torch.float).std() > 0.0
    return DynamicFilterOutput(keep=keep, reason=None if keep else f"zero_std_{round(rewards[0], 1)}")
```

### 1.6 训练-推理协同的设计权衡

**面试官可能问**：分离模式和共置模式各有什么优劣？如何选择？

**回答框架**：

**分离模式**：
- 优点：资源独立，互不干扰；易于扩展
- 缺点：需要权重同步，通信成本高；资源利用率可能低
- 适用场景：大规模训练，资源充足

**共置模式**：
- 优点：资源共享，利用率高；无需权重同步
- 缺点：显存竞争，需要精细管理；扩展性受限
- 适用场景：资源受限，小规模训练

**选择依据**：
- 模型大小：大模型通常需要分离模式
- 资源预算：资源有限时使用共置模式
- 训练规模：大规模训练使用分离模式

**结合 slime 代码**：
```python
# slime 的两种模式
# 分离模式
--actor-num-nodes 1
--actor-num-gpus-per-node 4
--rollout-num-gpus 4

# 共置模式
--colocate
--sglang-mem-fraction-static 0.8  # 调整 SGLang 显存占用
```

- 动态采样在训练初期效果更明显
- 后期可能收益递减

## 第三类：问题定位能力

> **考察重点**：排查问题的思路和解决方案

### 3.1 训练 loss 突然变成 NaN

**面试官可能问**：训练过程中 loss 突然变成 NaN，怎么排查？

**回答框架**：

**排查步骤**：
1. **检查数据**：是否有异常样本（如空序列、极长序列）
2. **检查学习率**：是否设置过大
3. **检查梯度范数**：是否有梯度爆炸
4. **检查 loss 计算**：是否有数值不稳定

**常见原因**：
- 学习率过大导致梯度爆炸
- 数据中存在异常值
- 某些 kernel 的数值不稳定

**解决方案**：
- 降低学习率
- 添加梯度裁剪
- 使用 `--no-check-for-nan-in-loss-and-grad` 跳过异常步骤

**结合 slime 代码**：
```python
# slime 的 NaN 处理
--no-check-for-nan-in-loss-and-grad  # 跳过 NaN 步骤

# 调试方法
# 1. 检查梯度范数
# 2. 检查 loss 值
# 3. 检查数据
```

### 3.2 推理生成乱码

**面试官可能问**：推理时生成乱码，可能是什么原因？

**回答框架**：

**排查步骤**：
1. **检查权重加载**：Megatron 是否正确加载 checkpoint
2. **检查权重同步**：参数是否正确转换和映射
3. **检查配置**：模型配置是否与 checkpoint 匹配

**常见原因**：
- checkpoint 路径错误
- 参数名称转换错误（如 PP 阶段的 layer ID）
- 某些 buffer 在释放过程中被意外清空

**解决方案**：
- 检查 Megatron 日志确认 checkpoint 加载
- 使用 `--check-weight-update-equal` 验证权重
- 使用 `--debug-rollout-only` 单独测试推理

**结合 slime 代码**：
```python
# slime 的调试方法
--debug-rollout-only  # 只运行推理

# 调试流程
# 1. 检查 Megatron 日志
# 2. 验证权重同步
# 3. 单独测试推理
```

### 3.3 训练速度突然变慢

**面试官可能问**：训练速度突然变慢，怎么定位瓶颈？

**回答框架**：

**排查步骤**：

**常见原因**：
- 某些样本过长（长尾问题）

**解决方案**：
- 使用 profiling 工具分析

### 3.4 模型能力突然下降

**面试官可能问**：训练一段时间后，模型能力突然下降，怎么排查？

**回答框架**：

**排查步骤**：
1. **检查数据质量**：是否有异常数据混入
2. **检查训练稳定性**：梯度范数、loss 曲线
3. **检查权重同步**：推理引擎是否收到正确权重
4. **检查评估方法**：评估数据是否变化

**常见原因**：
- 数据分布变化
- 训练不稳定（梯度爆炸）
- 权重同步失败
- 评估数据泄露

**解决方案**：
- 定期保存 checkpoint，回滚到正常状态
- 使用 debug replay 重现问题
- 检查数据管道

**结合 slime 代码**：

### 3.5 推理引擎 OOM

**面试官可能问**：推理引擎 OOM，怎么解决？

**回答框架**：

**排查步骤**：
1. **检查显存使用**：`nvidia-smi` 查看显存占用
2. **检查配置**：`--sglang-mem-fraction-static` 是否设置过大
3. **检查输入长度**：是否有极长输入

**常见原因**：
- 显存分配过大
- 输入序列过长
- 批量大小过大

**解决方案**：
- 降低 `--sglang-mem-fraction-static`
- 启用 FP8 KV cache
- 调整批量大小

**结合 slime 代码**：
```python

```


### 4.3 如何优化训练效率

**面试官可能问**：训练效率低，怎么优化？

**回答框架**：

**优化方向**：
1. **数据打包**：动态批处理，提高 GPU 利用率
2. **异步训练**：重叠训练和推理
3. **权重同步优化**：使用 Delta Sync
4. **显存优化**：启用重计算、卸载

**关键指标**：
- GPU 利用率：目标 > 90%
- 吞吐量：每秒处理的 token 数
- 训练时间：达到目标性能的时间

**优化策略**：
- 调整 `max_tokens_per_gpu` 优化数据打包
- 使用异步训练重叠计算
- 使用 Delta Sync 减少通信开销

**结合 slime 代码**：
```python
# slime 的效率优化
--use-dynamic-batch-size  # 动态批处理
--max-tokens-per-gpu 4608  # 最大 token 数
--update-weight-mode delta  # Delta Sync
--offload-train  # 训练时卸载
```

### 4.4 如何部署推理服务

**面试官可能问**：模型训练好后，怎么部署推理服务？

**回答框架**：

**部署架构**：
1. **推理引擎**：SGLang，支持高并发
2. **负载均衡**：Router，支持多实例
3. **监控**：Prometheus + Grafana

**关键配置**：
- 批量大小：平衡延迟和吞吐量
- 显存分配：`--sglang-mem-fraction-static`
- 并行策略：TP、PP 等

**性能优化**：
- FP8 量化：减少显存占用
- KV cache 优化：提高缓存命中率
- 批量推理：提高吞吐量

**结合 slime 代码**：
```python
# slime 的推理配置
--sglang-mem-fraction-static 0.8
--sglang-kv-cache-dtype fp8_e4m3
--rollout-num-gpus-per-engine 2  # TP size
```

### 4.5 如何保证数据质量

**面试官可能问**：怎么保证训练数据的质量？

**回答框架**：

**数据质量保障**：
1. **数据验证**：检查数据格式、长度、内容
2. **奖励过滤**：过滤低质量样本
3. **动态采样**：避免同质化数据
4. **数据监控**：监控数据分布变化

**关键指标**：
- 奖励分布：是否合理
- 数据多样性：是否有足够的变化
- 异常比例：异常样本的比例

**质量控制流程**：
- 训练前：数据验证、预处理
- 训练中：动态采样、奖励过滤
- 训练后：评估验证、问题定位

**结合 slime 代码**：
```python
# slime 的数据质量控制
--dynamic-sampling-filter-path slime.rollout.filter_hub.dynamic_sampling_filters.check_reward_nonzero_std
--rollout-sample-filter-path ...  # 自定义过滤函数

# 质量监控
# 1. 检查奖励分布
# 2. 统计数据多样性
# 3. 监控异常比例
```

## 第五类：业务与实际场景理解

> **考察重点**：场景价值、用户关心什么、上线成本、资源有限时的优先级

### 5.1 RL 后训练适合什么场景

**面试官可能问**：RL 后训练适合什么场景？不适合什么场景？

**回答框架**：

**适合场景**：
1. **推理任务**：数学、代码、逻辑推理
2. **Agent 任务**：工具使用、多轮交互
3. **对齐任务**：RLHF、偏好学习
4. **探索任务**：需要发现新解法的任务

**不适合场景**：
1. **简单任务**：SFT 就能解决的任务
2. **数据稀缺**：没有足够的奖励信号
3. **实时性要求高**：RL 训练周期长
4. **资源受限**：RL 需要大量计算资源

**选择依据**：
- 任务复杂度：复杂任务更需要 RL
- 奖励信号：需要可定义的奖励函数
- 计算预算：RL 需要大量计算
- 时间要求：RL 训练周期长

### 5.2 用户关心什么

**面试官可能问**：如果把这个系统给用户使用，用户最关心什么？

**回答框架**：

**用户关心的问题**：
1. **效果**：模型能力是否提升
2. **效率**：训练需要多长时间
3. **成本**：需要多少计算资源
4. **易用性**：使用是否简单
5. **稳定性**：训练是否稳定

**优化优先级**：
1. **效果优先**：首先保证模型能力提升
2. **效率其次**：优化训练效率
3. **成本控制**：在效果和效率之间平衡
4. **易用性**：提供清晰的文档和示例

**用户画像**：
- **算法研究员**：关心效果和灵活性
- **工程师**：关心效率和稳定性
- **产品经理**：关心成本和收益

### 5.3 上线成本有多高

**面试官可能问**：这个系统的上线成本有多高？

**回答框架**：

**成本构成**：
1. **计算成本**：GPU 训练时间
2. **存储成本**：checkpoint、数据
3. **网络成本**：权重同步、数据传输
4. **人力成本**：开发、维护

**成本估算**：
- 70B 模型：8xH100 训练 1 天约 $1000-2000
- 700B 模型：64xH100 训练 1 天约 $10000-20000
- Agent 训练：成本更高（需要工具调用）

**成本优化**：
- 使用 Delta Sync 减少通信成本
- 使用动态采样提高数据效率
- 使用共置模式减少资源需求

### 5.4 资源有限时优先优化什么

**面试官可能问**：如果资源有限，应该优先优化哪些部分？

**回答框架**：

**优化优先级**：
1. **数据质量**：高质量数据 > 大量数据
2. **算法选择**：选择合适的 RL 算法
3. **系统效率**：优化训练效率
4. **模型大小**：在效果和成本之间平衡

**具体建议**：
- **数据**：优先保证数据质量，使用动态采样
- **算法**：使用 GRPO 而不是 PPO（计算成本低 50%）
- **系统**：使用共置模式，减少资源需求
- **模型**：从小模型开始，验证有效性后再扩展

**资源分配**：
- 70% 数据和算法
- 20% 系统优化
- 10% 模型扩展

### 5.5 如何评估 ROI

**面试官可能问**：如何评估这个项目的 ROI？

**回答框架**：

**ROI 计算**：
```
ROI = (收益 - 成本) / 成本
```

**收益指标**：
- 模型能力提升：通过率、准确率
- 用户满意度：NPS、CSAT
- 业务指标：转化率、留存率

**成本指标**：
- 计算成本：GPU 时间
- 存储成本：数据、checkpoint
- 人力成本：开发、维护

**评估方法**：
1. **A/B 测试**：对比 RL 训练前后的效果
2. **成本效益分析**：计算每单位提升的成本
3. **长期价值**：考虑模型的复用价值

### 5.6 如何选择技术方案

**面试官可能问**：如果让你选择技术方案，你会怎么选？

**回答框架**：

**选择依据**：
1. **任务需求**：任务类型、复杂度
2. **资源限制**：计算、存储、网络
3. **团队能力**：技术栈、经验
4. **时间要求**：上线时间、迭代周期

**方案对比**：
| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| slime | 生产验证、灵活 | 学习曲线陡 | 大规模生产 |
| OpenRLHF | 易用、社区活跃 | 功能有限 | 快速验证 |
| veRL | 简单、轻量 | 扩展性差 | 小规模实验 |

**选择建议**：
- 大规模生产：选择 slime
- 快速验证：选择 OpenRLHF
- 小规模实验：选择 veRL

## 面试技巧总结

### 回答问题的框架

1. **STAR 法则**：
   - Situation：背景
   - Task：任务
   - Action：行动
   - Result：结果

2. **问题分析框架**：
   - 问题是什么
   - 为什么会出现
   - 如何解决
   - 效果如何

### 深挖问题的准备

1. **技术细节**：
   - 代码实现
   - 参数配置
   - 性能指标

2. **实验细节**：
   - 实验设计
   - 基线对比
   - 消融实验

3. **问题定位**：
   - 排查步骤
   - 常见原因
   - 解决方案

### 展示能力的技巧

1. **理论结合实践**：
   - 不仅讲理论，还要讲实现
   - 不仅讲结果，还要讲过程

2. **问题导向**：
   - 从问题出发，讲解决方案
   - 从需求出发，讲技术选型

3. **数据支撑**：
   - 用数据证明效果
   - 用对比证明价值
